## Цель

Наблюдать фактическое выполнение программы и изменения состояния без догадок.

## Теория

Debugger приостанавливает программу в выбранной точке и позволяет выполнять её
по шагам.

- `breakpoint` — точка остановки;
- `F5` — начать или продолжить отладку;
- `F10 / Step Over` — выполнить текущую строку и перейти к следующей;
- `Variables` — текущие значения доступных переменных.

Breakpoint не является точкой старта: программа выполняется от начала до него.

## Пошаговая модель

В цикле каждая итерация создаёт новое состояние:

1. `log_line` получает следующую строку;
2. увеличивается `line_count`;
3. условие может увеличить частный счётчик;
4. управление возвращается к следующей итерации.

После выхода из цикла строки закончились и счётчики больше не изменяются.

## Debugger и `print`

- Debugger полезен для локального пошагового анализа состояния.
- `print` удобен для быстрой гипотезы и временного наблюдения.
- В работающей системе вместо случайных `print` обычно применяют logging.

- При просмотре один раз перепутаны значения счётчиков.

## Math Core

Выполнение программы — дискретная последовательность состояний `S0, S1, S2...`.
Для счётчиков сохраняется `line_count >= cnc_count`.

## English Core

- **debugger** — отладчик;
- **breakpoint** — точка остановки;
- **step over** — выполнить текущую строку;
- **variables** — переменные;
- **current value** — текущее значение;
- **execution** — выполнение;
- **state** — состояние.

## Anki

2. **Что делает Step Over?** — Выполняет текущую строку и останавливается на следующей.
3. **Зачем окно Variables?** — Чтобы видеть реальное состояние данных на текущем шаге.
