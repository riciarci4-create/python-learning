## Цель

в одной небольшой программе.

## Архитектура

Арифметические функции выполняют одну операцию и не знают, какую команду выбрал
пользователь. Выбор операции остаётся во внешней управляющей логике.

```python
def add(a, b):
    return a + b
```

```python
if operation == "+":
    print(add(a, b))
```

Такое разделение позволяет вызывать и тестировать `add`, `subtract`, `multiply`
и `divide` независимо от интерфейса.

## Return contract

Первоначально при делении на ноль возвращалась строка. После code review контракт
стал устойчивее: `divide()` возвращает число или `None`, а пользовательское
сообщение формирует вызывающий код. Это не смешивает данные и интерфейсный текст.

## Микросценарии

- Корректное число и известная операция → числовой результат.
- Деление на ноль → `None`, затем понятное сообщение.
- Буквы вместо числа → `ValueError`, обработанный снаружи.
- Неизвестный оператор → отдельная ветвь `else`.

## Что было понято и исправлено

- Функция принимает аргументы и возвращает результат, не читая `operation` снаружи.
- `if operation == "+"` был ошибочно помещён внутрь арифметической функции.
- Исправлены пробелы PEP 8 и пустые строки между функциями.
- Во время review устранён повторный вызов `divide()` через переменную
  `division_result`.

## Math Core

- `add(a, b)` реализует `f(a, b) = a + b`.
- Деление определено при `b != 0`.
- Детерминированная арифметическая функция при одинаковых входах даёт одинаковый выход.

## English Core

## Anki

1. **Почему `add()` не должна знать про `operation`?** — Её ответственность только сложение переданных чисел.
2. **Что возвращает актуальная `divide(10, 0)`?** — `None`.
3. **Почему не стоит возвращать то число, то сообщение?** — Вызывающему коду трудно безопасно обрабатывать смешанные типы.
4. **Что делает внешний `if / elif / else`?** — Выбирает нужную операцию и формирует пользовательский ответ.
