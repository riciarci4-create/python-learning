- Ошибка: написал `age = "27"` в кавычках, поэтому возраст стал строкой `str`, а не числом `int`.

**2026-06-22 — Уроки 2–3: сравнение в условиях**

- Ошибка: несколько раз написал `=` вместо `==` при проверке условия.
- Причина: `=` присваивает значение, а `==` сравнивает два значения.
- Правильный вариант: в `if`, `elif` и фильтрации внутри циклов использовать `==`, если нужно проверить равенство.
- Паттерн для проверки: если строка начинается с `if` или `elif`, внутри условия почти всегда должно быть сравнение (`==`, `!=`, `>`, `<`, `>=`, `<=`), а не присваивание.

**2026-06-22 — Урок 4: бесконечный цикл**

- Ошибка: забыл менять переменную внутри `while`, из-за чего цикл стал бесконечным.
- Причина: `while` сам не двигает счётчик; условие должно когда-нибудь стать `False`.
- Правильный вариант: внутри цикла явно менять переменную, например `number += 1`.
- Паттерн для проверки: в каждом `while` найти переменную из условия и убедиться, что она изменяется внутри тела цикла.

## ПРОЕКТЫ НА GITHUB

> GitHub ведётся как инженерное портфолио, а не как архив случайных упражнений. Цель — постепенно показать будущим работодателям, заказчикам или партнёрам реальный рост навыков и способность доводить задачи до законченного результата.

**Зачем ведём GitHub:**

- фиксировать прогресс публично и проверяемо;
- учиться оформлять проекты как инженер, а не только писать код;
- собирать портфолио для будущей работы, фриланса и монетизации знаний;
- показывать ход мышления через README, структуру проекта, коммиты и разбор ошибок;
- дать потенциальному работодателю или заказчику возможность увидеть не обещания, а реальные артефакты.

**Правила публикации проектов:**

- публикуются только осмысленные задачи, мини-проекты и итоговые работы;
- у каждого проекта должен быть README: цель, запуск, пример результата, чему научился;
- коммиты должны быть регулярными и понятными;
- учебные ошибки не скрываются, но фиксируются как часть развития;
- каждый квартал выбирается 1 проект, который доводится до более аккуратного портфельного состояния.

**Структура GitHub-аккаунта:**

- один аккаунт на весь путь, не пересоздавать;
- отдельный репозиторий на каждый крупный проект;
- один репозиторий `learning-log` для заметок и мелких скриптов;
- 4–6 лучших проектов закрепляются как pinned repositories;
- профильный README кратко объясняет кто ты, что изучаешь и какие проекты уже есть.

**README крупного проекта обязан содержать:**

- одно предложение что делает проект;
- скриншот или GIF результата, если есть визуальный результат;
- команды запуска;
- технический стек;
- короткое описание структуры;
- что было сложно и как решено.

**Что нельзя делать:**

- публиковать незаконченный код без README как портфельный проект;
- коммитить реальные API-ключи, пароли, `.env`;
